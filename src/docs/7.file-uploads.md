# File Uploads 📎

All note tiers in SigNote support file attachments. Files are embedded directly into the note body using the rich text editor — images render inline, and other files appear as download cards.

---

## Attaching files

To attach a file, open a note for editing and either:

- **Drag and drop** a file anywhere onto the editor
- **Click the drop zone** at the bottom of the editor to open a file picker

Multiple files can be uploaded at once. Each file is uploaded immediately when added and embedded into the note at the cursor position.

---

## Supported file types

| Category  | Formats                   |
| --------- | ------------------------- |
| Images    | JPEG, PNG, GIF, WebP, SVG |
| Documents | PDF, DOCX, XLSX           |
| Text      | TXT, CSV, Markdown        |
| Archives  | ZIP                       |

Images render inline with a preview. All other files appear as a card showing the filename, type, and size, with a download button.

> Encrypted files (attached to Secrets or Seals) bypass the MIME type check — any file type can be uploaded when encryption is active.

---

## Limits

| Limit              | Value  |
| ------------------ | ------ |
| Max file size      | 5 MB   |
| Total storage/user | 100 MB |

---

## Encryption 🔐

File encryption follows the same model as note body encryption. The tier the file belongs to determines how it is handled:

### Tier 1 — Notes

Files are uploaded and stored as-is. No encryption is applied, matching the plaintext nature of the Notes tier.

### Tier 2 — Secrets

Before upload, the file bytes are encrypted client-side using **AES-GCM 256-bit** with a key derived from the MEK (the same `secretBodyKey` used for note bodies). A fresh random 12-byte IV is generated per file. Only the ciphertext is sent to the server — the server never sees the original file.

When you download or view the file, it is decrypted in the browser using the active session key.

### Tier 3 — Seals

Same client-side AES-GCM encryption as Secrets. The MEK must be unlocked for files to be uploaded or downloaded. Files are encrypted before leaving the browser and decrypted on demand.

In all encrypted cases, if the MEK is not available (encryption is locked), uploading a file to a Secret or Seal is blocked until you unlock.

---

## Storage

Files are stored in **AWS S3** or any S3-compatible provider (e.g. Cloudflare R2). The path structure is:

```
uploads/{userId}/{fileId}/filename      # unencrypted
uploads/{userId}/{fileId}/encrypted     # encrypted
```

See the [self-hosting guide](/docs/selfhosting-guide) for the required S3 environment variables.

---

## Soft delete

When a file is removed from a note (via the delete button on the attachment), it is soft-deleted: marked with a `deletedAt` timestamp in MongoDB. The actual S3 object is permanently deleted after a **1-hour grace period** by a background cleanup job.
