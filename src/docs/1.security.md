# Security 🛡️

SigNote is built around a simple promise: **the server cannot read what you don't want it to**. This page is the short tour of how that promise is kept — across sign-in, the three note tiers, your encryption profile, and what happens if you forget your passphrase.

If you want the implementation-level details, follow the links into [Encryption](/docs/encryption) and the [Threat model](/docs/threat-model).

---

## Why you can trust SigNote

Trust in a notes app should not come from marketing. It should come from properties that are **verifiable** by anyone who cares to look. SigNote is designed around a small set of those properties:

### 1. The server never sees your passphrase

Your passphrase is processed entirely in your browser. It is fed into **PBKDF2** with 600,000 iterations of SHA-256 to derive a 32-byte `deviceShare`. The passphrase itself is never sent to the server, never logged, and never written to disk.

### 2. The server never sees a key that can decrypt your notes

Encryption keys for Secrets and Seals are built from **two shares**:

```
MEK = deviceShare XOR serverShare
```

The server stores **only** `serverShare`. Without your passphrase, `serverShare` alone is useless — it cannot reconstruct the Master Encryption Key (MEK). The MEK only exists, briefly, in your browser's memory while you have the app open and unlocked.

### 3. The server never sees plaintext for Secrets and Seals

For Tier 2 (Secrets) and Tier 3 (Seals), the note body is encrypted with **AES-GCM 256-bit** in your browser **before** it leaves the device. The same is true for file attachments. The server stores ciphertext and an IV. That's it.

### 4. Your sign-in cannot be phished into another site

[Sign-In with Ethereum (SIWE)](/docs/sign-in) signs a message that is **cryptographically bound to the app's domain**. A signature obtained on `evil-signote.com` will be rejected by the real SigNote server.

### 5. Standard, audited primitives only

All cryptography uses the browser's built-in [Web Crypto API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Crypto_API): PBKDF2, HKDF, and AES-GCM. There are no custom crypto libraries, no hand-rolled ciphers, no "trust us" black boxes.

### 6. You can verify everything yourself

SigNote is open source. The code that runs in your browser, the code that runs on the server, and the code that handles every byte of your encrypted data — all of it is public. See the [Open source and verifiable](#open-source-and-verifiable) section below.

## Sign-in: who you are

SigNote supports two ways to prove your identity, and both avoid the worst password failure modes:

| Method                                          | What it is                                              | What the server gets                 |
| ----------------------------------------------- | ------------------------------------------------------- | ------------------------------------ |
| **Sign-In with Ethereum (SIWE)**                | You sign a domain-bound message with your wallet        | A signature and your wallet address  |
| **Google OAuth**                                | Standard OAuth 2.0 flow with Google                     | Email, name, profile picture         |

Key properties:

- **No passwords.** Neither method involves a password that SigNote stores or transmits.
- **No password reuse risk.** Wallet keys never leave your wallet; Google credentials never reach SigNote.
- **Domain-bound SIWE.** A signature for one origin cannot be replayed against another.
- **Sessions are short-lived.** JWT sessions are valid for 7 days, then expire.
- **Sign-in is separate from encryption.** Logging in does **not** unlock your Secrets or Seals — that requires your passphrase, every new tab.

→ Full breakdown: [Sign in](/docs/sign-in)

## The three tiers

Different notes have different sensitivity. Encrypting every grocery list is overkill, and putting a recovery phrase in a searchable plaintext field is dangerous. SigNote lets you pick the right level per note:

| Tier            | What the server stores                | Searchable           | Key model                          |
| --------------- | ------------------------------------- | -------------------- | ---------------------------------- |
| 1 — **Notes**   | Plaintext title + body                | Title and body       | None                               |
| 2 — **Secrets** | Plaintext title, ciphertext body      | Title only           | One shared session key (per user)  |
| 3 — **Seals**   | Plaintext title, ciphertext body, wrapped per-note key | Title only           | Unique per-note key, wrapped by MEK |

### Tier 1 — Notes (plaintext, by design)

Tier 1 notes are stored as plaintext. This is **intentional**: it enables full-text search across titles and bodies. Anything you put in a Tier 1 note is readable by anyone with database access — including you, and including the people running the server you use.

Use Tier 1 for everyday, non-sensitive content.

→ [Notes](/docs/notes)

### Tier 2 — Secrets (one session key)

Secrets are encrypted client-side with **AES-GCM 256-bit**, using a `secretBodyKey` derived from your MEK via HKDF (info string `signote-v1/secret-body`). After one unlock, all Secret bodies decrypt in the background — fast to browse.

Trade-off: a single key covers all your Secrets. If that key is ever leaked from your active session, every Secret is exposed.

→ [Secrets](/docs/secrets)

### Tier 3 — Seals (per-note keys)

Each Seal gets its **own** 32-byte Note Encryption Key (NEK), generated when the note is created. The body is encrypted with the NEK; the NEK is then wrapped with a `sealWrapKey` derived from your MEK **and the note ID**. Seals are never decrypted in bulk — you open one at a time.

The note ID is used as **Additional Authenticated Data (AAD)** on both the body and the NEK wrap. This cryptographically binds each ciphertext to its specific note: ciphertext from one Seal cannot be substituted into another.

A 60-second auto re-encryption timer also closes an open Seal if you walk away without editing.

→ [Seals](/docs/seals)

### File attachments

File attachments to Secrets and Seals are encrypted in the browser with the same AES-GCM keys as the note body, before upload. The server only ever stores encrypted bytes. → [File uploads](/docs/file-uploads)

### Tags and version history

[Tags](/docs/tags) are **plaintext metadata on every tier** — like titles, they are never encrypted, so don't put sensitive information in a tag name. [Version history](/docs/version-history) inherits the tier's encryption: for Secrets and Seals, version snapshots are stored as ciphertext only, encrypted with the same keys as the current note body.

## The encryption profile

The "encryption profile" is the small bundle of material the server needs to help you re-derive your MEK on a new device or new tab. It contains:

| Field         | What it is                                                              |
| ------------- | ----------------------------------------------------------------------- |
| `salt`        | 32 random bytes used in PBKDF2                                          |
| `kdf` params  | Hash, iteration count (600,000), output length                          |
| `serverShare` | 32 random bytes (one half of the MEK)                                   |
| `keyCheck`    | A known constant encrypted with a key derived from the MEK              |

The `keyCheck` is how the app knows whether you typed the right passphrase: it tries to decrypt `keyCheck` with the candidate MEK. Success means the passphrase was correct. Failure tells you nothing about the actual key — only that the attempt didn't work.

None of the items above can decrypt your notes on their own. They are useful only in combination with your passphrase, which only lives in your head.

→ [Encryption deep dive](/docs/encryption)

## Locks: when your data goes back into the safe

| Lock type    | Trigger                                                | What's cleared                                       | Re-prompts passphrase? |
| ------------ | ------------------------------------------------------ | ---------------------------------------------------- | ---------------------- |
| Soft lock    | Tab hidden / window minimised                          | MEK in memory                                        | No — uses `sessionStorage` |
| Hard lock    | Inactivity 5 min · sleep/resume · explicit Lock button | MEK + `deviceShare` in `sessionStorage`              | Yes                    |
| Tab close    | Tab/window closed                                      | MEK (tab dies) + `deviceShare` (`sessionStorage`)    | Yes (new tab)          |
| Sign-out     | You click sign out                                     | MEK + `deviceShare`                                  | Yes                    |

→ See the lock matrix in the [Threat model](/docs/threat-model).

## Recovery: what if you forget your passphrase?

This is where most encrypted-notes apps either lie ("we'll send you a reset email") or shrug ("sorry"). SigNote does neither.

### The honest part

Because the server cannot decrypt your notes, **the server cannot reset your passphrase**. There is no email-based password reset, no admin override, no backdoor. If your passphrase is the only thing on Earth that can produce your `deviceShare`, and you lose it, your `deviceShare` is lost too — and so is your MEK.

### The pragmatic part: a recovery file

SigNote offers an opt-in **recovery file** you can download after setting up your encryption profile. The file contains your `deviceShare` (base64-encoded), the schema type/version, your user ID, and a timestamp. With it, you can regain access even if you forget your passphrase.

Here's how recovery actually works:

1. You sign in normally.
2. You upload the recovery file at `/recover`.
3. The browser:
   - Reads your `deviceShare` from the file.
   - Fetches your `serverShare` from the server.
   - Reconstructs the MEK: `deviceShare XOR serverShare`.
   - Verifies the MEK against the server-stored `keyCheck` — if it doesn't match, the recovery file is for an old passphrase and is rejected.
4. You choose a **new** passphrase. The app derives a new `deviceShare`, computes a new `serverShare` so the MEK stays the same, and updates the encryption profile.

After recovery, your notes are unchanged — only the passphrase that unlocks them has been rotated.

### Why a recovery file isn't a backdoor

- It's **opt-in**. If you never download it, it doesn't exist.
- It contains only your `deviceShare` — exactly the share the server already does not have. Anyone who steals only the recovery file still cannot decrypt your notes: they would also need to sign in to your account and fetch `serverShare`.
- Conversely, anyone who only compromises the server still cannot decrypt your notes: they would also need your recovery file.
- The threat model becomes: an attacker needs **both** account access **and** the recovery file. This mirrors the original two-share design, just with the file substituting for memorised passphrase knowledge.

### How to store it

A recovery file is a **key**. Treat it like one:

- ✅ A reputable password manager.
- ✅ An encrypted drive or hardware key.
- ❌ A cloud notes app. ❌ Email. ❌ Chat apps.
- ❌ Any place that's covered by the same login as SigNote.

> ⚠️ If you change your passphrase, any older recovery file becomes invalid the moment it fails the `keyCheck` step. Re-download a fresh backup after passphrase rotation.

### Don't want a recovery file?

You don't have to download one. If you'd rather rely on memory alone — accepting that a forgotten passphrase means permanent loss of your encrypted data — that's a valid choice and SigNote supports it.

## Open source and verifiable

You don't have to take this page's word for any of it. SigNote is open source. **Every claim above is testable against the actual code** — and anyone with basic technical skills can do it.

👉 [github.com/irondsd/signote](https://github.com/irondsd/signote)

### What to look at if you want to verify

| Claim                                                       | Where to read the code                                          |
| ----------------------------------------------------------- | --------------------------------------------------------------- |
| All crypto uses the Web Crypto API, no custom libraries     | `src/lib/crypto.ts`                                             |
| Passphrase is hashed with PBKDF2 (600k iterations) locally  | `src/lib/crypto.ts` — `deriveDeviceShare`                       |
| MEK is `deviceShare XOR serverShare`, in memory only        | `src/lib/crypto.ts` — `xor32`, `importMEK`                      |
| The server never receives a plaintext Secret/Seal body      | `src/app/api/secrets/`, `src/app/api/seals/`                    |
| Per-note Seal keys are wrapped with a note-ID-derived key   | `src/lib/crypto.ts`, `src/controllers/` for seals               |
| SIWE rejects signatures with mismatched domain              | `src/lib/siwe.ts`, `src/config/auth.ts`                         |
| Recovery file structure and verification                    | `src/lib/recoveryBackup.ts`, `src/app/(main)/recover/page.tsx`  |
| Lock timers, sleep detection, sessionStorage handling       | `src/contexts/EncryptionContext.tsx`                            |

### Cheap ways to convince yourself

- **Open DevTools → Network** before saving a Secret. The request body is JSON containing an `iv` and `ciphertext`, never your plaintext.
- **Search the repo for "passphrase"**. You will not find it being sent in any network request — only being read from a local input.
- **Read `src/lib/crypto.ts` end to end.** It's a single file, intentionally small, and is the only place crypto happens.
- **Run it yourself.** The [self-hosting guide](/docs/selfhosting-guide) gets you a full local instance, so you can audit what's actually deployed.

### Trust, but verify — and improve

If you find a bug, a weakness, an unclear claim, or anything you want to challenge: open an issue or a pull request on GitHub. Security of this kind only improves when people read the code. Reading it is welcome and encouraged.
