# Encryption 🔐

This page is the full technical deep-dive into SigNote's encryption model. If you just want to understand what the tiers do, start with [Secrets](/docs/secrets) and [Seals](/docs/seals). If you want to know *how*, read on.

---

## Design goals

- The server stores only ciphertext — it can never read note bodies
- The passphrase is never stored, transmitted, or logged anywhere
- The Master Encryption Key (MEK) only exists in memory during an active unlocked session
- Losing your passphrase means losing access to encrypted notes — there's no recovery mechanism
- All cryptography uses the browser's built-in **Web Crypto API** — no third-party crypto libraries

---

## The two-share key architecture

The Master Encryption Key is never stored directly. Instead it's split into two shares:

```
MEK = deviceShare XOR serverShare
```

| Share | How it's made | Where it lives |
|-------|--------------|----------------|
| `deviceShare` | PBKDF2(passphrase, salt, 600k iterations) | `sessionStorage` (tab-scoped) |
| `serverShare` | Random 32 bytes generated at setup | Encrypted in MongoDB |

Neither share alone can reconstruct the MEK:

- **Server stolen** → attacker has `serverShare` but not `deviceShare` (which requires your passphrase). No MEK, no decryption.
- **`sessionStorage` read** → attacker has `deviceShare` but not `serverShare`. No MEK, no decryption.

Both shares are needed simultaneously. The MEK is reconstructed only in memory, as a non-extractable `CryptoKey` object, and only for the duration of an unlocked session.

→ See [Security model](/docs/security-model) for a full threat model.

## Passphrase derivation (PBKDF2)

During setup:

1. A random 32-byte `salt` is generated client-side and stored with your encryption profile
2. Your passphrase is run through **PBKDF2** to produce `deviceShare`:
   - Hash: SHA-256
   - Iterations: **600,000**
   - Output length: 32 bytes

When you unlock from a new tab or after closing `sessionStorage`, the same derivation runs again from your passphrase + the stored salt, reproducing the same `deviceShare`.

The high iteration count makes brute-forcing the passphrase computationally expensive.

> ⚠️ Your passphrase is the only thing protecting your data if your `serverShare` is ever stolen. Choose a strong, unique passphrase of at least 16 characters.

---

## Passphrase verification (keyCheck)

How does the app know if you entered the correct passphrase without storing it?

During setup, a `keyCheck` payload is created: a known constant is encrypted with a key derived from the MEK. When you unlock, the app attempts to decrypt `keyCheck`. If decryption succeeds, the passphrase was correct. If it fails, you're told and nothing is stored.

The `verifyKey` used for this is derived from the MEK via HKDF with info string `signote-v1/verify`.

---

## Key hierarchy (HKDF)

Once the MEK is reconstructed, it's imported as an HKDF base key. All working keys are derived from it using HKDF (SHA-256) with distinct `info` strings — this is called **domain separation** and ensures keys can't be confused with each other.

| Derived key | HKDF info string | Used for |
|-------------|-----------------|---------|
| `secretBodyKey` | `signote-v1/secret-body` | Encrypting and decrypting all Secret bodies |
| `verifyKey` | `signote-v1/verify` | Encrypting the `keyCheck` passphrase verification payload |
| `sealWrapKey(id)` | `signote-v1/seal-wrap/<noteId>` | Wrapping the per-note NEK for each Seal |

Because `sealWrapKey` includes the note ID in its info string, every Seal gets a unique wrapping key derived from the same MEK.

---

## Secrets encryption

Each Secret body is encrypted with `secretBodyKey`:

- **Algorithm:** AES-GCM 256-bit
- **IV:** 12 random bytes, freshly generated per encryption operation
- **Stored:** `{ iv, ciphertext }` in MongoDB (the body field is never plaintext)

All Secrets share the same session key, so after a single unlock they can all be decrypted at once in the background.

---

## Seals encryption — per-note keys

Each Seal uses a unique Note Encryption Key (NEK):

**Encryption:**

1. Generate a random 32-byte NEK
2. Encrypt the body with the NEK using AES-GCM 256-bit (note ID as AAD)
3. Derive `sealWrapKey` from MEK + note ID via HKDF
4. Wrap the NEK with `sealWrapKey` using AES-GCM 256-bit (note ID as AAD)
5. Store `encryptedBody` and `wrappedNoteKey` in MongoDB

**Decryption:**

1. Derive `sealWrapKey` from MEK + note ID (same as step 3 above)
2. Unwrap the NEK using `sealWrapKey`
3. Decrypt the body using the NEK

The note ID is used as **Additional Authenticated Data (AAD)** on both operations. This cryptographically binds the ciphertext to its note — if the note ID doesn't match during decryption, AES-GCM authentication fails.

---

## Cryptographic primitives

All operations use the browser's native [Web Crypto API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Crypto_API):

| Primitive | Used for |
|-----------|---------|
| **PBKDF2** (SHA-256, 600k iterations) | Passphrase → `deviceShare` |
| **HKDF** (SHA-256) | MEK → working keys |
| **AES-GCM 256-bit** | All encryption and decryption |
| **XOR** | Combining two 32-byte shares into the MEK |

Keys are created as **non-extractable** `CryptoKey` objects wherever possible. This prevents JavaScript from reading the raw key bytes, even from the same page.

---

## sessionStorage and tab lifecycle

`deviceShare` is kept in `sessionStorage` after the first unlock. This allows the MEK to be silently reconstructed on page reload — you don't need to re-enter your passphrase every time you refresh.

`sessionStorage` is **tab-scoped**: it's automatically cleared when you close the tab. The MEK cannot be reconstructed from a new tab without re-entering the passphrase.

Explicit lock also clears `sessionStorage` immediately, along with the in-memory MEK and `CryptoKey` objects.

---

## What the server never sees

- Your passphrase (never transmitted)
- The MEK (never leaves the browser)
- `deviceShare` (never sent server-side)
- Note Encryption Keys (NEKs) in unwrapped form
- Any plaintext note body for Secrets or Seals

→ See [Secrets](/docs/secrets) and [Seals](/docs/seals) for per-tier details.
