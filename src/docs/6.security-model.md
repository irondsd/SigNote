# Security model 🛡️

This page describes what SigNote is designed to protect against, where the trade-offs are, and what it explicitly does *not* promise. Understanding the model helps you make informed decisions about what to store and at what tier.

---

## What SigNote protects

### Server compromise

If an attacker gains full read access to the MongoDB database, they get:

- ✅ Notes tier 1 — **plaintext** (fully readable)
- 🔐 Secrets and Seals titles — **plaintext** (readable, by design)
- 🔒 Secrets and Seals bodies — **ciphertext only** (not readable without the MEK)
- 🔑 `serverShare` — present in the database, but **useless without `deviceShare`**

Without the user's passphrase, `serverShare` alone cannot reconstruct the MEK. The attacker would need to brute-force the passphrase against the PBKDF2 derivation — 600,000 iterations of SHA-256 makes this expensive.

→ See [Encryption](/docs/encryption) for key architecture details.

### Client-side `sessionStorage` read

If an attacker can read the browser's `sessionStorage` (e.g. via XSS), they get `deviceShare`. But without `serverShare` from the server, they still can't reconstruct the MEK.

Note: SigNote sanitises rendered HTML (via DOMPurify) to reduce XSS surface area in note content.

### Active session memory

During an unlocked session, the MEK exists in memory as a non-extractable `CryptoKey` object. JavaScript code on the page cannot read the raw key bytes — but other scripts running in the same origin could potentially interact with the Web Crypto API using that key.

This is the inherent trade-off of any client-side encryption model: the key must exist in memory for encryption and decryption to happen.

## Lock events

| Event | What's cleared | Passphrase to re-unlock? |
|-------|---------------|--------------------------|
| Tab hidden (soft lock) | MEK (memory) | No — `deviceShare` kept in `sessionStorage` |
| Inactivity 5 min (hard lock) | MEK (memory) + `deviceShare` (`sessionStorage`) | Yes |
| Sleep / resume gap >30s (hard lock) | MEK (memory) + `deviceShare` (`sessionStorage`) | Yes |
| Explicit Lock button (hard lock) | MEK (memory) + `deviceShare` (`sessionStorage`) | Yes |
| Tab close | `deviceShare` (`sessionStorage`) — MEK already gone with the tab | Yes (new tab) |
| Sign-out | MEK (memory) + `deviceShare` (`sessionStorage`) | Yes |

After a hard lock or sign-out, encrypted notes are inaccessible until the passphrase is entered again. After a soft lock, returning to the tab re-unlocks silently using the retained `deviceShare`.

## SIWE domain binding

Sign-In with Ethereum validates the domain embedded in the signed message against the server's configured `NEXTAUTH_URL`. A valid signature from `evil-signote.com` will be rejected when submitted to the real app's API.

This prevents a class of phishing attacks where an attacker captures a SIWE signature from a lookalike site and replays it.

## Known trade-offs and limitations

### Tier 1 notes are intentionally unencrypted

[Notes](/docs/notes) are stored as plaintext. This enables full-text search across titles and bodies, which is the point. If you store sensitive information there, it's visible to anyone with database access.

Use [Secrets](/docs/secrets) or [Seals](/docs/seals) for anything you wouldn't want read from the database.

### Titles are always plaintext

Across all three tiers, note **titles** are stored unencrypted. This enables search and navigation. If your titles themselves contain sensitive information, be aware they're visible to the server.

### Passphrase strength matters

The security of your encrypted notes depends on your passphrase. If someone has your `serverShare` (from the database) and can guess or brute-force your passphrase, they can reconstruct the MEK.

Choose a long, unique passphrase. The minimum is 16 characters. More is better.

### No recovery mechanism

If you forget your passphrase, your encrypted notes are permanently inaccessible. There is no password reset, no recovery key, no backdoor. This is by design — a recovery mechanism would be a weakness.

### `sessionStorage` persists during the browser session

Between page reloads in the same tab, `deviceShare` stays in `sessionStorage`. This means you don't have to re-enter your passphrase on every refresh, but it also means `deviceShare` is present in the browser for the entire tab session.

Closing the tab clears it automatically.

## What SigNote does not protect against

- An attacker who has **both** your `serverShare` and your passphrase
- An attacker with full access to your device during an active, unlocked session
- A compromised browser extension with access to `sessionStorage` and network requests
- Side-channel attacks on the Web Crypto API implementation
- Metadata: note count, creation time, title length, and access patterns are visible to the server

---

## Summary

SigNote's encryption is designed so that neither the server alone nor the client alone is sufficient to decrypt your notes. The passphrase never leaves your device, and the MEK never leaves your browser. If you use strong passphrases and understand the tier model, your encrypted notes are well-protected against database-level compromise.

As with any cryptography-heavy application, treat this as evolving software and review the implementation carefully before relying on it for real-world sensitive data.
